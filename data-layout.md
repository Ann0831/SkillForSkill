# Skill Detector Data Directory Layout

## Default Root Directory

The default data root directory for `skill_candidate_detector` is:

```txt
~/.SkillForSkillStorage
```

This path must stay consistent with `defaultDataDir` in `config.js`. When later code needs to read or write detector-related data, it should start from this root directory and then enter the appropriate subfolder based on the data's purpose.

The work task-count threshold is configured by `workTaskLimit` in `config.json`. When the number of task Markdown files under a given `work/<work-type>/` is greater than this threshold, and `state.json` has not yet marked that a potential skill has been produced, that work type should be considered a candidate for generating a `potential_skill`.

## Directory Structure

```txt
~/.SkillForSkillStorage/
  work/
    <work-type>/
      description.md
      state.json
      <task>.md
      ...
  discarded_work/
    <work-type>/
      description.md
      state.json
      <task>.md
      ...
  potential_skill/
    <skill>.md
  discarded_skill/
    <skill>.md
```

## `work/`

`work/` stores the work types produced or analyzed by the agent.

Each work type uses one subfolder, for example:

```txt
~/.SkillForSkillStorage/work/debugging/
~/.SkillForSkillStorage/work/refactor/
~/.SkillForSkillStorage/work/test-writing/
```

Each work type folder should include a `description.md` that describes what this kind of work generally does and which similar tasks can be placed in this work.

Each work type folder should also include a `state.json` that stores the current processing state for that work type:

```json
{
  "hasPotentialSkill": false,
  "isSkillCandidate": null
}
```

`hasPotentialSkill` indicates whether this work type has already been turned into a potential skill. The default value is `false`.

`isSkillCandidate` indicates whether this work type is suitable for conversion into a skill. The default value is `null`, meaning it has not yet been judged. Later code may update it to `true` or `false` based on agent or system judgment.

A work type folder may contain multiple task Markdown files, for example:

```txt
~/.SkillForSkillStorage/work/debugging/description.md
~/.SkillForSkillStorage/work/debugging/state.json
~/.SkillForSkillStorage/work/debugging/task-001.md
~/.SkillForSkillStorage/work/debugging/task-002.md
```

These files represent observed work samples, classification results, statistics, or other evidence needed for later skill-candidate judgment.

## `discarded_work/`

`discarded_work/` stores work type data that has been judged unsuitable for preservation as a skill.

When a `work/<work-type>/` is judged as `isSkillCandidate: false`, or the agent or system determines that it is not suitable for conversion into a skill, the entire work type folder may be moved to:

```txt
~/.SkillForSkillStorage/discarded_work/<work-type>/
```

The contents of `discarded_work/<work-type>/` keep the original `work/<work-type>/` format and may include `description.md`, `state.json`, and multiple task Markdown files. After the move, the work type no longer lives under `work/`, so it should not appear in work-list results that read only `work/`.

After moving to `discarded_work/`, `state.json` must include `isFutureWorkUnneeded`, which records whether the user believes similar future tasks no longer need handling:

```json
{
  "hasPotentialSkill": false,
  "isSkillCandidate": false,
  "isFutureWorkUnneeded": true
}
```

`isFutureWorkUnneeded` is a boolean. `true` means the user believes this kind of future task no longer needs to be handled; `false` means the work is unsuitable for a skill, but similar future tasks may still need handling.

## `potential_skill/`

`potential_skill/` stores recommended skill data generated by the agent.

When a work type appears frequently and the system judges that it may be worth preserving as a skill, later code can create a Markdown file for the recommended skill, for example:

```txt
~/.SkillForSkillStorage/potential_skill/error-debugging-helper.md
~/.SkillForSkillStorage/potential_skill/refactor-planner.md
~/.SkillForSkillStorage/potential_skill/test-case-writer.md
```

These files represent skill candidates that have not yet been formally installed or adopted. They are not published skills.

## `discarded_skill/`

`discarded_skill/` stores potential skills rejected by the user.

When a `potential_skill/<skill>.md` is rejected by the user, that Markdown file may be moved to:

```txt
~/.SkillForSkillStorage/discarded_skill/<skill>.md
```

`discarded_skill/` stores the rejected skill candidate itself for later auditing or review. After the move, the skill no longer lives under `potential_skill/`, so it should not be treated as a potential skill still awaiting user confirmation.

## Relationship Between `work` and `potential_skill`

`work/<work-type>` is the storage location for original work types and observation data.

`discarded_work/<work-type>` is the storage location for work types judged unsuitable for conversion into skills. It preserves the original work samples for later auditing or review.

`potential_skill/<skill>.md` is the location for a recommended skill generated by the agent after work type accumulation.

`discarded_skill/<skill>.md` is the location for a potential skill after the user rejects it.

Later code should first accumulate work data under `work`, then decide based on rules or thresholds whether to generate a corresponding `potential_skill`.

When deciding whether a work type should be written as a potential skill, count the `.md` task files under `work/<work-type>/`, excluding `state.json`. If the number of tasks is greater than `workTaskLimit` in `config.json` and `hasPotentialSkill` is `false`, the work type has reached the condition for generating a potential skill.

For now, this layout only defines the required state fields in `work/<work-type>/state.json` and the basic storage locations for `work` task files, `discarded_work` task folders, `potential_skill` files, and `discarded_skill` files. Other file formats are intentionally not locked down yet, to avoid constraining later implementation details too early.
