# Agent Operations Guide

This document defines how an agent should use `SkillForSkill` after completing a meaningful user task: record the task, classify the work, check skill candidates, and handle potential skills based on the user's response.

## Required First Step: Find SkillForSkill Server

For every completed meaningful task, the agent must first find a usable SkillForSkill server URL. Recording happens after this step. Do not skip server discovery just because the task may later match a storage skip condition.

Reading this skill is not sufficient. The first observable action for a completed meaningful task must be to run the scoped SkillForSkill client with `ensure-server`. The client health checks the fallback server URLs, starts the server if none are available, health checks again, and returns the valid URL.

Only after the scoped client returns a usable server URL should the agent record the task unless a storage skip condition applies.

Existing Codex sessions may need a new thread or restart before they load this stronger instruction.

## Trigger Timing

Whenever an agent completes a meaningful task requested by the user, it must first find the SkillForSkill server, then enter the storage-recording workflow unless a storage skip condition applies.

Do not run the server discovery or storage-recording workflow in these cases:

- The interaction is pure conversation or produced no meaningful task output.
- The user explicitly asked not to use SkillForSkill or not to record the task.
- The task is not complete.

## Recording Criteria

After finding a usable server URL, the agent must record every completed meaningful task. A task does not need to be reusable, repeatable, or likely to become a skill before it is recorded.

The agent must not record the task and must not add a work or task only when one of these storage skip conditions applies. These skip criteria apply only to storage recording; they do not skip the required server discovery step for completed meaningful tasks.

- The user explicitly asked not to record it.
- The task produced no meaningful output beyond conversation.
- The task is incomplete.

Otherwise, the agent should continue with the task-recording workflow. If it does not record the task, it should keep the skip reason clear.

## REST Server

When this document is generated by the installer, it records the absolute project directory, the REST fallback URL contract, and the data storage location. The placeholders in the source document are replaced by the install script.

Project directory:

```txt
__SKILLFORSKILL_PROJECT_DIR__
```

REST fallback URLs:

```txt
__SKILLFORSKILL_SERVER_URLS__
```

Data is stored by default in:

```txt
__SKILLFORSKILL_DATA_DIR__
```

Agents should use the scoped SkillForSkill client for normal server discovery and command calls:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" ensure-server
```

Expected response:

```json
{
  "ok": true,
  "url": "http://127.0.0.1:3000",
  "host": "127.0.0.1",
  "port": 3000,
  "started": false
}
```

Agents should request persistent approval for this narrow command prefix when the environment asks for authorization:

```txt
node __SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js
```

The client only talks to the local fallback URLs from the server contract and only uses `/api/health`, `/api/read`, and `/api/write`.

If `ensure-server` returns `ok: true`, the server is already running or was started successfully. Treat the returned `url` as the selected SkillForSkill server URL for this task.

If the health check fails, install dependencies if needed from the project directory:

```bash
cd "__SKILLFORSKILL_PROJECT_DIR__" && npm install
```

Then run `ensure-server` again.

Raw REST health checks are available for manual debugging:

```bash
curl -sS "<candidate-url>/api/health"
```

Expected raw health response:

```json
{
  "ok": true,
  "service": "SkillForSkill",
  "mode": "rest",
  "version": "0.0.0"
}
```

If `ensure-server` still fails, assume the port is wrong, the server did not start, or another service owns every contract port.

If the execution environment requires authorization for `npm install`, the scoped client, starting the server, or writes to `__SKILLFORSKILL_DATA_DIR__`, the agent should request the required authorization and then continue using the same project directory.

## REST Command Format

SkillForSkill exposes exactly two command endpoints:

| Method | Endpoint | Command type |
| --- | --- | --- |
| GET | `${SELECTED_SKILLFORSKILL_SERVER_URL}/api/read` | Read commands |
| POST | `${SELECTED_SKILLFORSKILL_SERVER_URL}/api/write` | Write commands |

Both endpoints use a JSON request body:

```json
{
  "command": "list-works",
  "args": {}
}
```

For browser-style reads and user-space pages, `GET /api/read` also accepts query parameters:

```txt
${SELECTED_SKILLFORSKILL_SERVER_URL}/api/read?command=list-works&args=%7B%22includeFutureUnneededDiscardedWork%22%3Atrue%7D
```

`args` must be URL-encoded JSON and defaults to `{}` when omitted.

Agents should normally call these endpoints through the scoped client:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" read list-works '{"includeFutureUnneededDiscardedWork":true}'
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" write add-work '{"workName":"<workName>","description":"<description>"}'
```

Successful responses use:

```json
{
  "ok": true,
  "result": []
}
```

Error responses use:

```json
{
  "ok": false,
  "error": "message"
}
```

This workflow may only use commands that currently exist:

- Read commands:
  - `list-works`
  - `list-work-candidates`
  - `list-potential-skills`
  - `get-work-skill-reference`
  - `get-work`
  - `get-task`
  - `get-potential-skill`
- Write commands:
  - `add-work`
  - `add-task`
  - `add-potential-skill`
  - `discard-work`

There is currently no "install skill" command and no "update work state" command. The agent must not invent commands that do not exist.

## Task Recording Workflow

After passing the inclusion criteria, the agent must first read all active work and any discarded work that the user has marked as unnecessary for future prompts:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" read list-works '{"includeFutureUnneededDiscardedWork":true}'
```

If the current task matches a returned work with `isDiscarded: true` and `isFutureWorkUnneeded: true`, the user has previously chosen not to be prompted about skills for this kind of task in the future. In that case, the agent must not add a task, must not create a potential skill, and must not prompt the user again to install this kind of skill.

If no such discarded work matches, the agent must check whether the current task belongs to an existing active work where `isDiscarded: false`. When deciding, the agent must use each work's `description` and confirm that the current task matches the type of work and similar tasks described there.

If the task matches an existing work, add the task directly:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" write add-task '{"workName":"<workName>","taskName":"<taskName>","content":"<taskContent>"}'
```

If no existing work is suitable, add a new work first:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" write add-work '{"workName":"<workName>","description":"<description>"}'
```

The `description` must clearly explain what this kind of work does and which similar tasks belong in this work.

Then add the task:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" write add-task '{"workName":"<workName>","taskName":"<taskName>","content":"<taskContent>"}'
```

The `taskName` must be stable and avoid collisions, such as by using a timestamp or short slug. If `add-task` fails because a task with the same name already exists, the agent must retry with a new `taskName` and must not overwrite the existing task.

The `taskContent` must include at least:

- A summary of the user's original request.
- What the agent actually completed.
- The main files or commands involved.
- Why the task belongs to this work, including how it relates to the work `description`.

## Skill Candidate Check

After adding a task, the agent must list candidate work that has exceeded the threshold and does not yet have a potential skill:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" read list-work-candidates '{}'
```

This command uses `workTaskLimit` from `config.json` and returns active work matching these conditions:

- `taskCount > workTaskLimit`
- `hasPotentialSkill` is `false`

If the returned array is empty, the agent does not need to evaluate skill candidates.

If candidates are returned, the agent must evaluate only the first candidate work in the result to decide whether it is worth turning into a skill. Other candidate work should be left for future task completions.

Before evaluating the first candidate work, the agent must read the full skill reference content for that work:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" read get-work-skill-reference '{"workName":"<workName>"}'
```

The agent must use the returned `referenceText` to decide whether this work is worth turning into a skill. If it is worth doing, `referenceText` should also be the primary reference material for creating the potential skill Markdown.

If the first candidate work is not worth turning into a skill, the agent must move it to `discarded_work` and mark that similar future tasks may still need to be handled:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" write discard-work '{"workName":"<workName>","isFutureWorkUnneeded":false}'
```

If it is worth turning into a skill, the agent must create a potential skill:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" write add-potential-skill '{"workName":"<workName>","skillName":"<skillName>","content":"<skillMarkdown>"}'
```

After this command succeeds, it marks `hasPotentialSkill` as `true` in `work/<workName>/state.json`, preventing that work from appearing again in `list-work-candidates`.

The `skillMarkdown` should describe the recommended skill and include at least:

- The skill name.
- The skill description.
- The recurring task type this skill is meant to solve.
- The referenced work name and a summary of sample tasks.

## User Confirmation Workflow

After creating a potential skill, the agent must ask the user whether they want to install it.

`potential_skill/<skillName>.md` is only a candidate skill draft and must not be treated as an installed skill.

If the user agrees to install it, the agent must convert the candidate Markdown into the formal skill directory structure and then hand it off to the external skill installation mechanism. This project currently has no command for installing a skill.

To locate candidate Markdown drafts before conversion, the agent may list potential skill draft paths:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" read list-potential-skills '{}'
```

A formal skill needs at least:

```txt
<skillName>/
  SKILL.md
```

`SKILL.md` must include YAML frontmatter with at least:

```md
---
name: <skillName>
description: <when this skill should be used>
---
```

Therefore, after the user agrees, the workflow is: read `potential_skill/<skillName>.md`, convert it into a valid formal `SKILL.md`, place it in a `<skillName>/` directory, and then pass it to the external installation flow.

If the user rejects the skill but does not say that future prompts for this kind of task are unnecessary, the agent must move the work to `discarded_work` and mark that similar future tasks may still need to be handled:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" write discard-work '{"workName":"<workName>","isFutureWorkUnneeded":false}'
```

If the user clicks or says that they do not want future skill prompts for this kind of task, the agent must move the work to `discarded_work` and mark that similar future task prompts are unnecessary:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" write discard-work '{"workName":"<workName>","isFutureWorkUnneeded":true}'
```

Later, when the agent runs the task-recording workflow again, it must use:

```bash
node "__SKILLFORSKILL_PROJECT_DIR__/scripts/skillforskill-client.js" read list-works '{"includeFutureUnneededDiscardedWork":true}'
```

to identify work where `isDiscarded: true` and `isFutureWorkUnneeded: true`, avoiding future task additions or install prompts for the same kind of skill.

## Judgment Principles

When classifying work, the agent should choose the semantically closest work category, rather than classifying only by filename, tool, or surface-level traits of a single task.

When creating new work, `workName` should be short, stable, descriptive, and must avoid path separators.

When evaluating whether a work is worth turning into a skill, the agent should confirm that the work is repeatable, generalizable, and likely to benefit from a fixed workflow. If it is a one-off request, depends too heavily on specific context, or has no clear process to preserve, do not create a potential skill for now.
