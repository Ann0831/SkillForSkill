# Agent Operations Guide

This document defines how an agent should use `SkillForSkill` after completing a user task: record the task, classify the work, check skill candidates, and handle potential skills based on the user's response.

## Trigger Timing

Whenever an agent is working on a meaningful task requested by the user, it must decide after completion whether the task should enter this workflow.

Do not run this workflow in these cases:

- The interaction is pure conversation or produced no substantive task output.
- The user explicitly asked not to record the task.
- The task is not complete.

## Inclusion Criteria

After completing the task, the agent must first evaluate whether this task could recur in the future and whether it has a chance to become a skill.

If the task matches any of the following cases, the agent must not record it and must not add a work or task:

- It is clearly a one-off request.
- It depends too heavily on the current conversation, a specific file state, or short-lived context.
- It has no repeatable workflow.
- Even if it recurs, a skill would be unlikely to improve the process meaningfully.

Only when the agent judges that the task may recur and could become a skill in the future should it continue with the task-recording workflow.

## Available Commands

When this document is generated by the installer, it records the absolute `main.js` path on the current machine and the data storage location. The placeholders in the source document are replaced by the install script.

All commands must be executed with the absolute `main.js` path generated during installation. They must not depend on the current shell working directory:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- <command> [--key value]
```

Data is stored by default in `__SKILLFORSKILL_DATA_DIR__`. The first time a command needs to write to this directory, if the execution environment requires filesystem authorization, the agent should use an escalated command and suggest that the user allow the command prefix `node __SKILLFORSKILL_MAIN_JS__` once. All later SkillForSkill commands must continue using the same absolute `main.js` path to avoid repeated authorization requests caused by different working directories.

This workflow may only use commands that currently exist:

- `list-works`
- `list-potential-skills`
- `add-work`
- `add-task`
- `add-potential-skill`
- `discard-work`

There is currently no "install skill" command and no "update work state" command. The agent must not invent commands that do not exist.

## Task Recording Workflow

After passing the inclusion criteria, the agent must first read all active work and any discarded work that the user has marked as unnecessary for future prompts:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- list-works --includeFutureUnneededDiscardedWork true
```

If the current task matches a returned work with `isDiscarded: true` and `isFutureWorkUnneeded: true`, the user has previously chosen not to be prompted about skills for this kind of task in the future. In that case, the agent must not add a task, must not create a potential skill, and must not prompt the user again to install this kind of skill.

If no such discarded work matches, the agent must check whether the current task belongs to an existing active work where `isDiscarded: false`. When deciding, the agent must use each work's `description` and confirm that the current task matches the type of work and similar tasks described there.

If the task matches an existing work, add the task directly:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- add-task --workName <workName> --taskName <taskName> --content "<taskContent>"
```

If no existing work is suitable, add a new work first:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- add-work --workName <workName> --description "<description>"
```

The `description` must clearly explain what this kind of work does and which similar tasks belong in this work.

Then add the task:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- add-task --workName <workName> --taskName <taskName> --content "<taskContent>"
```

The `taskName` must be stable and avoid collisions, such as by using a timestamp or short slug. If `add-task` fails because a task with the same name already exists, the agent must retry with a new `taskName` and must not overwrite the existing task.

The `taskContent` must include at least:

- A summary of the user's original request.
- What the agent actually completed.
- The main files or commands involved.
- Why the task belongs to this work, including how it relates to the work `description`.

## Skill Candidate Check

After adding a task, the agent must list candidate work that has exceeded the threshold and does not yet have a potential skill:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- list-work-candidates
```

This command uses `workTaskLimit` from `config.json` and returns active work matching these conditions:

- `taskCount > workTaskLimit`
- `hasPotentialSkill` is `false`

If the returned array is empty, the agent does not need to evaluate skill candidates.

If candidates are returned, the agent must evaluate only the first candidate work in the result to decide whether it is worth turning into a skill. Other candidate work should be left for future task completions.

Before evaluating the first candidate work, the agent must read the full skill reference content for that work:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- get-work-skill-reference --workName <workName>
```

The agent must use the returned `referenceText` to decide whether this work is worth turning into a skill. If it is worth doing, `referenceText` should also be the primary reference material for creating the potential skill Markdown.

If the first candidate work is not worth turning into a skill, the agent must move it to `discarded_work` and mark that similar future tasks may still need to be handled:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- discard-work --workName <workName> --isFutureWorkUnneeded false
```

If it is worth turning into a skill, the agent must create a potential skill:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- add-potential-skill --workName <workName> --skillName <skillName> --content '<skillMarkdown>'
```

After this command succeeds, it marks `hasPotentialSkill` as `true` in `work/<workName>/state.json`, preventing that work from appearing again in `list-work-candidates`.

The `skillMarkdown` should describe the recommended skill and include at least:

- The skill name.
- The skill description.
- The recurring task type this skill is meant to solve.
- The referenced work name and a summary of sample tasks.

## User Confirmation Workflow

After creating a potential skill, the agent must ask the user whether they want to install it.

`potential_skill/<skillName>.md` is only a candidate skill draft and must not be treated as an installed skill.

If the user agrees to install it, the agent must convert the candidate Markdown into the formal skill directory structure and then hand it off to the external skill installation mechanism. This project currently has no command for installing a skill.

To locate candidate Markdown drafts before conversion, the agent may list potential skill draft paths:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- list-potential-skills
```

A formal skill needs at least:

```txt
<skillName>/
  SKILL.md
```

`SKILL.md` must include YAML frontmatter with at least:

```md
---
name: <skillName>
description: <when this skill should be used>
---
```

Therefore, after the user agrees, the workflow is: read `potential_skill/<skillName>.md`, convert it into a valid formal `SKILL.md`, place it in a `<skillName>/` directory, and then pass it to the external installation flow.

If the user rejects the skill but does not say that future prompts for this kind of task are unnecessary, the agent must move the work to `discarded_work` and mark that similar future tasks may still need to be handled:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- discard-work --workName <workName> --isFutureWorkUnneeded false
```

If the user clicks or says that they do not want future skill prompts for this kind of task, the agent must move the work to `discarded_work` and mark that similar future task prompts are unnecessary:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- discard-work --workName <workName> --isFutureWorkUnneeded true
```

Later, when the agent runs the task-recording workflow again, it must use:

```bash
node __SKILLFORSKILL_MAIN_JS__ --- list-works --includeFutureUnneededDiscardedWork true
```

to identify work where `isDiscarded: true` and `isFutureWorkUnneeded: true`, avoiding future task additions or install prompts for the same kind of skill.

## Judgment Principles

When classifying work, the agent should choose the semantically closest work that represents the recurring work pattern, rather than classifying only by filename, tool, or surface-level traits of a single task.

When creating new work, `workName` should be short, stable, reusable, and must avoid path separators.

When evaluating whether a work is worth turning into a skill, the agent should confirm that the work is repeatable, generalizable, and likely to benefit from a fixed workflow. If it is a one-off request, depends too heavily on specific context, or has no clear process to preserve, do not create a potential skill for now.
